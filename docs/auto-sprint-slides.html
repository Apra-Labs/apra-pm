Inputs vs /pm

You provide three things. auto-sprint does the rest.

With /pm you use commands like plan, start, cleanup etc.
With /auto-sprint you hand over the keys.

🎯
Epic IDs
Your existing beads epics — e.g. gh-toy-4ef, gh-toy-7rp. Same epics you'd pass to /pm.
🌿
Branch
feat/four-p1-v2 — created automatically if it doesn't exist yet.
🏁
Goal (optional)
Exit when P1, P1/P2, or P1/P2/P3 beads issues are all closed. Default: P1/P2.

Architecture

Four phases, fully autonomous

The same agents you know from /pm — planner, doer, reviewer — plus plan-reviewer, deployer, integ-test-runner, and harvester. auto-sprint wires them together and loops until the goal is met.

Getting started

If you've used /pm, you already know the pieces

auto-sprint drives them for you. You don't need to know the JSON — just describe what you want.

Plain English
"Use /auto-sprint on epics BD-1 and BD-2, branch feat/my-sprint, close when P1 and P2 are done"
↓ LLM constructs
LLM builds this
/auto-sprint {"issues":["BD-1","BD-2"],"branch":"feat/my-sprint","goal":"P1/P2"}
goal P1 · P1/P2 · P1/P2/P3
max_cycles default 5
base_branch default main
requirementsFile optional path

Issue tracker

My issues are in Azure DevOps

Use LLM + az cli to pull PBIs/Features from ADO into beads, then hand beads to auto-sprint.

Bridge pattern
  • Run az boards work-item list → ask LLM to create matching beads epics
  • Pass those beads epics to auto-sprint — sprint loop polls beads for exit-gate signals
  • ADO stays source of truth; auto-sprint never needs an ADO token
requirements.md
  • Dump ADO acceptance criteria, design notes, or constraints into requirements.md
  • Pass it via requirementsFile — planner reads it before making any decisions
  • No API access needed — paste, copy, or generate the file however you like
Preparation

Documents to prepare in advance

One is required. The rest unlock additional phases.

📋
beads epics
At least one epic in beads describing what to build. One line is enough — the planner expands it.
📄
requirements.md optional
Detailed requirements, constraints, or design decisions. Passed via requirementsFile.
🚀
deploy.md
How to build, deploy, and smoke-test. Activates the integration test phase.
🧪
integ-test-playbook.md
Setup / reset / teardown for the test environment. Paired with deploy.md to activate Phase 3.
